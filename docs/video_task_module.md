# 任务创建模块说明

## 1. 模块目标

当前模块对应系统里的第一块后端能力：`创建视频任务`。

它主要解决两个紧密相关的问题：

1. 当用户提交商品标题、卖点、目标平台、视频时长、风格以及最小上传素材信息后，系统如何生成一条合法、可追踪的任务记录。
2. 当任务创建成功后，系统如何把任务推进到“工作流已自动启动”的占位状态。

这个模块当前仍然不负责：

- API 路由
- 文件持久化存储
- 素材处理
- 剧本生成
- 分镜生成
- 后台渲染

## 3. 模块边界

当前模块保留四类核心对象：

### 3.1 `TaskStatus`

任务状态枚举。

当前只保留最小状态集，避免在第一阶段把状态机做复杂。

### 3.2 `CreateVideoTaskCommand`

创建任务时的输入对象。

它和 `VideoTask` 不合并，原因是两者语义不同：

- `CreateVideoTaskCommand` 表示“外部传进来的原始创建请求”
- `VideoTask` 表示“系统内部已经创建成功的任务实体”

保留这层边界后，后续接 API、表单、消息队列都会更清楚。

### 3.3 `VideoTask`

系统内部任务实体。

它代表已经通过校验并进入系统管理范围的任务记录，后续可以继续附加：

- 素材信息
- 剧本信息
- 分镜信息
- 渲染记录

当前还额外承载了两个最小工作流字段：

- `workflow_stage`
- `workflow_message`
- `workflow_result`

这些字段的作用是让“任务创建成功”“工作流开始执行”“草稿计划生成完成”之间有一个清晰的可见过渡。

### 3.4 `UploadedAsset`

上传素材的最小元数据对象。

当前只保留：

- 文件名
- 内容类型

这里单独保留成对象是合理的，因为它已经不是普通字符串，而是后续素材模块会继续扩展的独立概念。

### 3.5 `InMemoryTaskRepository`

当前阶段的临时仓储实现。

保留仓储对象的原因不是“为了设计模式”，而是为了给后续替换数据库留稳定接口。当前阶段先用内存仓储，可以把注意力放在业务结构上。

### 3.6 `start_task_workflow(...)`

这是当前阶段新增的工作流启动函数。

它不接真实 Agent，也不接真实模型，只做一件事：

- 把任务从“已创建”推进到“工作流已自动启动”的占位状态

保留这一步的意义是：

- 明确任务和工作流是两个内部步骤
- 让页面和后续模块可以围绕工作流阶段继续扩展
- 为后续真实剧本生成、分镜生成留出清晰入口

### 3.7 `finish_task_workflow(...)`

这是当前阶段用于保存 `agent/` 工作流输出的函数。

它只接收普通字典，不直接依赖工作流模块，原因是：

- 任务模块只负责保存任务状态和结果
- 工作流模块负责怎么生成结果
- 两边通过普通结构化数据连接，边界更清楚

## 4. 为什么删掉 `VideoTaskService`

上一版里有一个 `VideoTaskService` 类，但当前逻辑只有三步：

1. 校验输入
2. 创建任务实体
3. 保存任务

这层逻辑太薄，单独包成类会让抽象层数多于实际复杂度。

因此当前版本改成：

- `create_video_task(...)`：模块级函数，负责主流程
- `_validate_create_command(...)`：私有校验函数，负责输入规则

这样的好处：

- 阅读路径更短
- 调用方式更直接
- 当前阶段更容易修改
- 后续如果逻辑真的变重，再升格成服务类也不迟

## 5. 当前模块的设计原则

后续写每个模块时，都按下面几个问题先判断：

1. 这个类是否真的代表一个独立概念？
2. 这层抽象是否降低了复杂度，而不是制造复杂度？
3. 现在就拆分，是否是为了当下问题，而不是为了未来想象？
4. 如果只剩一个方法，这一层是否更适合用函数表达？
5. 这个对象未来是否明确会被替换，例如内存仓储替换成数据库仓储？

## 6. 后续扩展方向

这个模块后面最自然的扩展顺序是：

1. 接 FastAPI 路由，把 `CreateVideoTaskCommand` 接进接口层
2. 替换内存仓储为数据库仓储
3. 增加更多任务状态和工作流阶段
4. 把任务创建和后续素材处理链路连接起来
5. 把占位工作流替换成真实剧本生成入口
6. 将 `workflow_result` 从内存字段迁移到数据库或任务结果表

## 7. 维护规则

从当前模块开始，后续每新增一个模块，都同步维护一份对应说明文档，至少包含：

- 模块目标
- 模块边界
- 保留了哪些对象
- 删除了哪些不必要抽象
- 为什么这样拆分
