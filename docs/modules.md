# 系统模块说明

本文档维护当前已完成的各个模块，描述功能、原理和关键设计决策。

---

## agent/asset_preprocessor.py — 素材预处理管线

**功能**：对用户上传的图片做统一修复和标准化，确保进入下游流水线的素材质量一致。

**处理流程**：

- 清晰度估计：用拉普拉斯方差判断图像锐度，低于阈值做 UnsharpMask 锐化
- 曝光修正：计算平均亮度，过暗（<40）提亮，过曝（>220）压暗
- 白平衡校正：取画面最亮 5% 像素作为白点参考，对各通道线性缩放使参考点趋近纯白
- 画幅适配：根据目标宽高比居中裁剪为 9:16
- 输出缩放到 720×1280，JPEG 质量 92

**设计原则**：完全确定性，不依赖 LLM。不根据质量做分级——不管原图多差，统一修到可用状态。

## agent/video_generation_workflow.py — 工作流编排

**功能**：把视频任务从素材分析推进到最终成片的完整编排。

**核心流程**：

- 素材分析：调用素材预处理管线，再调多模态 LLM 对每张图做角色标注（hook / 展示 / 特写 / 场景 / CTA）
- 导演决策：LLM 从预置策略库中选择创意策略，生成候选版本和素材建议
- 商品上下文：整理商品信息、卖点、风格约束为结构化上下文
- 剧本生成：输出 hook + body + CTA + tone，附带范例引导，审核不通过时重试
- 分镜设计：拆解为 3-4 个分镜，每个分镜决定 render_strategy（image_to_video / text_to_video / ai_image_then_video / crop_and_ken_burns），运动描述用精确物理语言
- 素材匹配：将可用素材分配到各分镜
- 创作计划：汇总为下游可消费的执行计划
- 视频渲染：调用 Seedance 或本地预览
- 最终检查：规则校验 + 质量评估

**降级策略**：LLM 全部不可用时（AIGC_DISABLE_LLM=1），剧本和分镜走规则模板兜底。

## agent/seedance_video_renderer.py — Seedance 视频渲染

**功能**：调用火山方舟 Seedance API 生成分镜视频，拼接、叠加字幕。

**关键能力**：

- 并行渲染：所有分镜通过 ThreadPoolExecutor 同时提交和轮询，而非串行等待
- 分镜 prompt 工程：将分镜描述转换为模型可理解的 prompt，注入物理约束和画质参数
- 轮询机制：可配置的超时（默认 360s）和间隔（默认 10s），处理异步任务等待
- 网络容错：DNS pinning 绕过 WSL2 慢 DNS，5xx 错误自动重试 3 次
- 视频拼接：ffmpeg concat demuxer 合并分镜片段
- 字幕叠加：Pillow 逐帧绘制中文字幕。Seedance 本身不支持中文渲染，在 prompt 中要求模型不生成任何文字，字幕完全由程序在后期叠加上去。使用 Noto Sans CJK 字体，半透明底加白色文字

## agent/simple_video_renderer.py — 本地预览渲染

**功能**：当 Seedance 不可用或失败时，用 Pillow + imageio 生成本地预览视频作为兜底。

**原理**：根据分镜描述和字幕文本，逐帧绘制静态画面并合成 MP4。画面质量远不如 Seedance，但保证链路完整可用。

## agent/asset_preprocessor.py — 素材预处理

**功能**：对用户上传图片做清晰度、曝光、白平衡和画幅的自动修复。

**处理流程**：
- 拉普拉斯方差估计锐度 → 不够锐的做 UnsharpMask
- 平均亮度检测 → 过暗提亮、过曝压暗
- 白点参考法白平衡校正
- 居中裁剪为 9:16 后缩放到 720×1280

**设计决策**：不走分级标注路线（清晰的用、模糊的不用），而是统一修复。原因是用户没有替代素材，标注了"模糊不适合"也没用，不如修好再用。

## video_task_module.py — 领域模型

**功能**：定义任务状态、数据结构、仓储接口和工作流状态管理。不依赖网络框架。

**核心实体**：
- TaskStatus：QUEUED → PROCESSING → COMPLETED / NEEDS_REVIEW / FAILED / CANCELLED
- VideoTask：任务实体，包含参数、进度、事件和结果
- InMemoryTaskRepository：内存仓储

**设计决策**：不用 ORM 和持久化数据库，比赛阶段内存存储够用，且避免了数据库配置复杂度。

## task_creation_demo_app.py — Web 应用

**功能**：FastAPI 服务端渲染的 Web 界面。表单提交 → 后台线程启动工作流 → 轮询展示结果。

**路由**：
- GET / → 任务创建表单
- POST /tasks → 接收表单和文件，创建任务，启动后台工作流
- GET /tasks/{id} → 详情页，每 3 秒轮询刷新
- /uploads/ → 静态文件服务

**设计决策**：不用 React 前后端分离，FastAPI SSR 足够展示，比赛评委更关心"能跑起来看到效果"。
