# 视频生成工作流模块规划

## 1. 模块目标

这个模块放在 `agent/` 目录下，负责从“任务已创建”继续推进到“视频草稿计划已生成”。

当前阶段先不使用 LangChain / LangGraph，而是用普通 Python 函数组成轻量工作流。这样可以先把输入、输出、状态和页面展示跑清楚，后续如果分支、重试、人工干预变复杂，再迁移到框架。

## 2. 工作流边界

输入：

- 任务 ID
- 商品标题
- 商品卖点
- 目标平台
- 视频时长
- 风格预设
- 用户自定义风格期望
- 上传素材元数据

输出：

- 素材分析结果
- 商品上下文
- 剧本规划
- 分镜列表
- 素材匹配结果
- 创作计划
- 本地 MP4 预览结果
- 最终检查结果
- 工作流状态和阶段说明

## 3. 步骤设计

### 3.0 Director 决策

作用：
- 在进入剧本生成前，先判断这个商品更适合哪种创意策略
- 生成 3 个候选创意版本，形成轻量 A/B 对比
- 选择一个当前最适合渲染的主版本，并说明选择原因
- 发现素材不足时给出补充建议，而不是直接静默降级

实现方式：
- 优先使用 LLM 生成结构化 JSON
- LLM 不可用时使用规则兜底
- 当前阶段只渲染被选中的一个版本，候选版本先用于评审展示和后续扩展

Director 输出字段：
- `selected_strategy`：最终采用的策略
- `selected_reason`：为什么选它
- `candidate_variants`：候选版本列表，每个版本包含风格、Hook 角度、风险、预估分
- `asset_advice`：素材补充建议
- `render_advice`：渲染建议和降级策略

这样做的意义是让工作流不再只是“固定顺序调用 API”，而是在创作前有一个可解释的决策层。

### 3.1 素材规则校验

作用：

- 判断文件类型是否支持
- 判断是否有可用图片或视频
- 生成基础素材清单

实现方式：

- 规则代码

原因：

- 文件名、Content-Type、数量校验是确定性逻辑，不应该交给 LLM。

### 3.2 素材语义理解

作用：

- 判断素材里大概是什么商品
- 判断素材适合首帧、细节展示还是场景展示
- 识别素材质量问题和风险

实现方式：

- 如果上传文件已经保存到本地，则读取图片并调用火山方舟多模态模型
- 如果没有真实文件内容或模型未配置，则返回明确的降级结果

原因：

- “素材用途建议”属于视觉理解和判断，不能只靠规则拍脑袋。

### 3.2.1 素材画像与商品身份卡

作用：

- 把上传素材从“文件列表”升级为“素材画像”
- 识别哪些素材适合作为商品外观锚点、细节参考或直接视频片段
- 生成 Product Identity Card，作为后续剧本、分镜和视频 prompt 的共同约束

当前输出：

- `asset_profiles`：每个素材的角色、质量评分、适用镜头和风险说明
- `product_identity_card`：商品类型、外观摘要、主色调、关键部件、必须保持项、禁止变化项
- `motion_affordance`：商品允许动作、风险动作和禁止动作
- `standardized_file_path`：经过清晰度、曝光、白平衡和画幅修复的原图，保留给多模态理解
- `anchor_file_path`：使用 `rembg` 抠出商品主体后放入统一棚拍背景的锚点图，优先供渲染使用
- `shared_scene_background_path`：不包含商品主体的统一墙面和桌面底图，供开场承接镜使用

实现说明：

- 优先调用 LLM 生成结构化身份卡
- LLM 不可用时根据商品标题、卖点和素材基础信息生成保守身份卡
- 身份卡不会编造品牌、Logo 或型号，只描述已知或可保守推断的商品事实
- 抠图结果会经过透明度质量门槛检查；主体过度透明时不会直接进入视频渲染
- 同一批素材只要存在一张抠图合格的商品锚点，其他抠图失败的图片仍可用于多模态分析，但渲染时会复用合格锚点，避免原始拍摄背景突然插入成片

### 3.3 商品上下文结构化

作用：

- 把标题、卖点、平台、风格、素材分析结果整理成统一上下文
- 为剧本和分镜提供稳定输入

实现方式：

- LLM 生成结构化 JSON
- 规则代码兜底

### 3.4 剧本规划

作用：

- 生成 15 秒带货视频的叙事结构
- 包含 Hook、卖点展开、CTA 和语气风格

实现方式：

- LLM

原因：

- 剧本文案和叙事策略是创作任务，需要模型参与。

审核与打回：

- 生成后立刻做规则审核
- 检查是否存在 `hook`、`body`、`cta`、`tone`
- 如果 LLM 返回不可解析 JSON 或字段缺失，则把问题传回剧本生成节点重做
- 当前最多重试 2 次，仍失败则进入人工确认状态

### 3.5 分镜生成

作用：

- 把剧本拆成可执行镜头
- 每个镜头包含时长、画面说明、字幕、旁白、素材需求
- 每个镜头都必须描述状态转移：开始画面、核心动作、结束画面
- 不把所有镜头都写成商品展示。15 秒视频至少包含一个剧情场景、一个商品揭示或商品展示、一个明确收尾

实现方式：

- LLM 生成
- 规则代码检查总时长

叙事角色：

- `hook`：用具体场景、痛点或反差抓住注意力，可以不出现商品
- `problem`：展示用户问题，例如桌面拥挤、通勤负担、收纳困难，可以不出现商品
- `context`：建立使用环境，例如宿舍、办公室、厨房、户外，可以不出现商品
- `product_reveal`：从剧情场景切换到真实商品，优先使用统一背景上的商品锚点淡入
- `feature_demo`：展示核心卖点，必须绑定真实商品素材
- `detail_proof`：用材质、结构、Logo 或局部细节证明可信度，必须绑定真实商品素材
- `lifestyle_result`：展示使用后的结果或氛围，不要求突出可识别商品主体
- `cta`：回到真实商品画面，完成行动引导

审核与打回：

- 分镜的 `purpose` 和 `duration_seconds` 由 LLM 判断
- 代码审核字段完整性和总时长
- 分镜必须包含 `narrative_role`、`scene_goal`、`initial_state`、`action`、`final_state`、`camera_motion`
- 分镜不能全部是同一张商品图的轻微运动；至少一个镜头承担剧情铺垫或结果表达
- 商品揭示镜和商品事实镜要分开：前者负责空间承接，后者负责真实外观和卖点证明
- 如果分镜缺字段、时长无效或总时长超限，则只打回分镜节点重做
- 不重新执行素材分析、商品上下文和剧本生成，除非后续审核发现问题源头来自上游

这样做的目标是减少“动态 PPT”式输出，让每个镜头都能回答：

- 这个镜头为什么存在
- 开始是什么状态
- 中间发生什么动作
- 结束变成什么状态
- 这个变化如何服务商品卖点

### 3.6 素材匹配

作用：

- 为每个分镜选择渲染策略
- 有合适图片时选择 `image_to_video`
- 没有图片但画面可描述时选择 `text_to_video`
- 强依赖真实商品画面但素材不足时选择 `needs_user_asset`
- `product_reveal` 使用共享空棚拍底图和真实商品锚点，在本地生成淡入桥接镜

实现方式：

- 当前先用规则 + 素材摘要
- 后续接 Embedding 检索和多模态相似度

说明：

- 不要求每个分镜都必须绑定一张用户上传图片
- 素材少时允许部分镜头走文生视频
- 分镜时长仍然使用 LLM 生成的 `duration_seconds`
- `hook/problem/context/lifestyle_result` 默认保留为独立剧情镜，不会因为存在商品锚点就被替换成棚拍图
- 只有 `product_reveal` 或显式声明 `continuity_mode=shared_scene_bridge` 的镜头才使用统一棚拍桥接

### 3.7 素材缺口补全

作用：

- 明确识别哪些分镜没有合适素材
- 避免把少量商品图片硬套到所有分镜，导致视频像 PPT 轮播
- 把素材缺口转换成当前系统可执行的渲染策略

实现方式：

- `image_to_video`：真实商品外观、细节、包装等镜头优先复用上传商品图
- `text_to_video`：氛围、场景、转场类镜头可以只用画面描述生成
- `ai_image_then_video`：当前项目还没有明确文生图 endpoint，所以先交给 Seedance 文生视频生成，并保留补全 prompt
- `needs_user_asset`：强依赖真实商品但没有素材时，优先复用商品图；完全没有商品图时也交给视频模型生成缺失画面，同时标记真实性风险

说明：

- 当前实现不假装已经接入文生图模型
- 页面会展示每个素材缺口的原始策略、最终策略、补全说明和真实性风险
- 素材缺口不会阻断视频生成；评审和用户可以先看到完整视频，再判断是否需要补素材重生成
- 后续如果接入文生图，只需要把 `text_to_video_prompt` 升级成“先生成场景图，再图生视频”

### 3.8 创作计划生成

作用：

- 生成后续渲染需要的执行计划
- 包含画面、字幕、旁白、BGM、转场、是否需要图生视频

实现方式：

- LLM 生成结构化草稿
- 规则代码补齐状态字段

### 3.9 本地预览视频渲染

作用：

- 视频渲染优先调用 Seedance
- 共享空背景镜使用本地稳定渲染，不交给 Seedance 自由补全
- 如果视频模型失败或未配置，则回退到本地 MP4 预览
- 让用户先看到完整链路效果，并知道是否发生了回退

实现方式：

- `agent/seedance_video_renderer.py`
- `agent/simple_video_renderer.py`
- 使用 `Pillow` 生成 9:16 竖版帧
- 使用 `imageio` / `imageio-ffmpeg` 写出 MP4

说明：

- `image_to_video` 分镜会把上传图片传给 Seedance
- `text_to_video` 分镜只使用画面描述生成视频
- Seedance prompt 会注入商品身份卡、动作能力、状态转移字段和禁止变化项
- 开场承接镜会从 `shared_scene_background_path` 开始，在同一背景中逐渐淡入真实商品锚点，不调用 Seedance，避免模型擅自新增桌面、家具或错误商品
- 剧情场景镜仍可调用 Seedance 文生视频，用来提供痛点、环境和结果画面；这类镜头禁止突出可识别商品和品牌标识
- 各分镜拼接使用约 `0.45s` 的逐帧交叉叠化，不再使用硬切或中间黑帧
- 如果 Seedance 失败，页面会展示回退原因
- 本地 fallback 视频仍然是静态图 / 文字占位 + 字幕，不代表真实视频模型效果

当前限制：

- `rembg` 是通用抠图模型。复杂角度、遮挡和深色商品可能只保留部分主体轮廓
- 统一背景策略解决的是跨镜头背景跳变，不等于已经完全解决商品主体抠图质量
- 后续需要增加主体完整性评分，并允许用户在页面上选择或替换商品锚点图

### 3.10 最终检查

作用：

- 检查总时长是否合规
- 检查每个分镜是否有画面方案
- 检查预览视频是否生成成功
- 检查是否存在明显缺失或失败步骤

实现方式：

- 规则代码

## 4. 打回策略

当前实现的是受控打回，不让 LLM 自己决定流程。

流程：

1. 生成节点产出结果
2. 审核节点检查结果
3. 通过则继续下一步
4. 不通过且可重试，则只重跑当前节点
5. 重试时把上一次审核问题传回生成节点
6. 超过重试次数后进入 `needs_review`

当前自动打回节点：

- `script_plan`
- `storyboard`

暂不自动打回的节点：

- 素材处理：目前还没保存真实文件内容，复杂打回价值不高
- 素材匹配：当前还是简单规则匹配
- 创作计划：当前还没接真实渲染，先用最终检查兜底

工作流结果会记录：

- `script_review`
- `storyboard_review`
- `review_attempts`

页面会展示每次尝试是否通过，以及失败后执行的是 `retry`、`continue` 还是 `needs_user_review`。

本地测试时如果不想触发真实模型调用，可以设置 `AIGC_DISABLE_LLM=1`。设置后工作流会走降级逻辑，不调用 DeepSeek 或火山方舟。

## 5. 进度回调与阶段耗时

`run_video_generation_workflow(...)` 当前支持传入 `progress_callback(stage, message, progress)`。
任务详情页会通过这个回调拿到阶段进度，并展示在页面上。

当前进度节点：

1. `asset_analysis`：素材分析
2. `product_context`：商品上下文整理
3. `script_plan`：剧本规划与审核
4. `storyboard`：分镜生成与审核
5. `asset_matching`：分镜素材匹配
6. `asset_gap_completion`：素材缺口补全决策
7. `creation_plan`：创作计划生成
8. `render_video`：视频模型渲染或本地回退
9. `final_check`：最终检查

控制台会在每个阶段完成时输出该阶段耗时，例如：

```text
[video_generation_workflow] 素材处理完成：task_id=task_xxx, elapsed=0.42s
[video_generation_workflow] 视频渲染完成：task_id=task_xxx, elapsed=38.10s
[video_generation_workflow] 工作流执行结束：task_id=task_xxx, workflow_status=completed, workflow_stage=draft_ready, total_elapsed=45.23s
```

这样做的意义是：

- 用户在页面上能看到当前执行到哪一步
- 开发时能从控制台判断慢在 LLM、Seedance 还是本地渲染
- 后续接数据库任务表时，可以直接把这些事件落库

Seedance 子任务也会输出分镜级诊断信息：

- `shot_index`：当前是哪一个分镜
- `seedance_task_id`：火山方舟返回的异步任务 ID
- `elapsed`：该分镜从提交到完成/失败的耗时
- `last_status`：超时时最后一次轮询到的任务状态

如果再次出现超时，需要优先保留 `seedance_task_id`。
有了这个 ID 才能区分是本地网络轮询中断、服务端排队过久，还是服务端任务最终失败。

## 6. 字幕生成与叠加策略

字幕不交给 Seedance 生成。
分镜节点仍然让 LLM 生成每个 shot 的 `subtitle`，但 Seedance prompt 会明确要求画面中不要出现任何文字。
Seedance 只负责生成无字画面，视频合并后再由本地后处理根据分镜时间轴把字幕烧录进去。

这样做的原因是：

- 视频模型生成可读文字不稳定，中文尤其容易变成乱码或伪字符
- 分镜字幕是结构化字段，后处理叠加更可控
- 后续做多语言字幕时，只需要替换 `subtitle` 字段，不需要重新生成画面

当前实现会生成：

- `seedance_raw.mp4`：Seedance 生成并合并后的无字幕版本
- `seedance_final.mp4`：按分镜时间轴烧录字幕后的最终视频

字幕后处理优先用 Pillow 逐帧绘制，并直接加载中文字体文件，避免 WSL 的 fontconfig 找不到中文字体。
字体查找顺序：

- `AIGC_SUBTITLE_FONT_PATH`
- `/usr/share/fonts/opentype/noto/NotoSansCJK-Regular.ttc`
- `/usr/share/fonts/truetype/wqy/wqy-microhei.ttc`
- `/mnt/c/Windows/Fonts/msyh.ttc`
- `/mnt/c/Windows/Fonts/simhei.ttf`

如果这些字体都不存在，字幕叠加会明确失败并保留无字幕视频。

## 7. 当前不做的内容

- 不做自动发布
- 不做复杂多 Agent 状态机
- 不做素材向量库
- 不做长期任务队列
- 不把 LLM 决策结果直接用于文件路径或系统命令

## 8. 类和函数取舍

当前优先使用函数，不提前创建很多类。

保留函数边界：

- `run_video_generation_workflow(...)`
- `process_assets(...)`
- `build_product_context(...)`
- `plan_script(...)`
- `build_storyboard(...)`
- `match_assets_to_storyboard(...)`
- `build_creation_plan(...)`
- `run_final_check(...)`

暂时不创建 `AssetAgent`、`ScriptAgent`、`RenderAgent` 这类类名，因为当前每一步更像 workflow step，而不是独立自治智能体。
